# 오목 게임 — 개발 계획서 (Development Plan)

> 범위: **2인 로컬 대전 오목** / 단일 HTML / 의존성 0
> 전략: **점진적 개발** — 매 단계가 "실행 가능한 산출물"을 남기고, 작은 단위로 검증하며 쌓아 올린다.

---

## 1. 개발 전략

1. **MVP 우선:** FR-01~06(Must)만으로 "둘 수 있는 게임"을 먼저 완성한다.
2. **단계별 실행 가능:** 각 단계 종료 시 브라우저에서 눈으로 확인 가능한 결과가 있어야 한다.
3. **규칙 먼저 테스트:** 5목 판정은 버그가 잦은 부분 → 단계 M3에서 케이스로 검증.
4. **확장은 나중:** AI·금수·온라인은 본 계획 외(로드맵 §7).

---

## 2. 마일스톤 (단계별 계획)

### M0. 프로젝트 셋업
- **목표:** 빈 캔버스가 뜨는 `index.html` 골격
- **작업:** HTML 골격, `<canvas>`, 상태바·버튼 영역, 모듈 빈 객체(GameState/Rules/Renderer/Game/main) 선언
- **산출물:** 실행되는 빈 페이지
- **검증:** 브라우저에서 캔버스 영역과 버튼이 보인다
- **DoD:** 콘솔 에러 없이 페이지 로드

### M1. 보드 렌더링 (FR-01)
- **목표:** 15×15 바둑판 그리기
- **작업:** `Renderer.drawBoard()` — 격자선, 화점 5개, 여백/칸 크기 계산
- **산출물:** 정적 바둑판
- **검증:** 격자 15×15, 화점 위치 정확, 교차점 간격 균일
- **DoD:** 보드가 정사각으로 반듯하게 렌더

### M2. 착수 + 턴 (FR-02, FR-03)
- **목표:** 클릭으로 돌 놓기 + 흑/백 교대
- **작업:** `Game.handleClick`(픽셀→격자 변환), `Rules.isValidMove`, `GameState.board` 갱신, `Renderer.drawStones`, 상태바 차례 갱신
- **산출물:** 두 명이 번갈아 돌 놓기 가능
- **검증:** 교차점 정확히 스냅, 점유칸/보드밖 클릭 무시, 차례 표시 토글
- **DoD:** FR-02·03 인수 기준 충족

### M3. 승리 판정 (FR-04)
- **목표:** 5목 감지
- **작업:** `Rules.checkWin(board,x,y,player)` — 4방향 양방향 카운트, `winLine` 수집
- **산출물:** 5목 시 콘솔/상태로 승리 감지
- **검증:** §4 테스트 케이스 전부 통과(가로/세로/↘/↗/경계/장목)
- **DoD:** 모든 방향·경계에서 정확히 판정

### M4. 종료 처리 + 재시작 (FR-05, FR-06)
- **목표:** 승리 연출 + 입력 차단 + 리셋
- **작업:** `status` 전이, 승자 메시지, `Renderer.drawWinLine`(승리 5돌 강조), 종료 시 클릭 차단, `Game.reset()` + 버튼 바인딩
- **산출물:** 완결된 한 판 → 재시작 가능 = **MVP 완성**
- **검증:** 승리 후 착수 불가, 승자 표시, 다시 시작 시 초기화
- **DoD:** §8 MVP 인수 기준 전체 통과

### M5. 부가 기능 (FR-07, FR-08, FR-09)
- **목표:** 무승부 / 마지막 수 표시 / 무르기
- **작업:** `Rules.isDraw`, `Renderer.highlightLast`, `Game.undo`(moveHistory pop + 차례 복원) + 버튼
- **산출물:** 사용성 향상판
- **검증:** 보드 가득→무승부, 최근 수 표식, 무르기 후 상태/화면 일치
- **DoD:** 각 기능 독립 동작, 회귀 없음

### M6. 다듬기 (선택)
- **목표:** 마감 품질
- **작업:** 호버 미리보기(FR-11), 수순 표시(FR-10), 반응형 캔버스 크기, 색/여백 정리
- **검증:** 다양한 창 크기에서 깨짐 없음
- **DoD:** 시각적 완성도 확보

## 3. 작업 순서 / 의존성

```
M0 ──► M1 ──► M2 ──► M3 ──► M4 ──► (MVP 완성) ──► M5 ──► M6
셋업    보드    착수    판정   종료/재시작           부가     다듬기
              └ M2는 M1(렌더) 필요
                     └ M3는 M2(상태) 필요
                            └ M4는 M3(판정) 필요
```
M5·M6는 MVP(M4) 이후 독립적으로 추가 가능.

## 4. 검증 전략 — 5목 판정 테스트 케이스

`Rules.checkWin`은 가장 버그가 잦으므로 아래 케이스를 명시적으로 확인한다.

| # | 상황 | 입력 | 기대 |
|---|---|---|---|
| T1 | 가로 5목 | (4,7)~(8,7) 흑 | 승리 |
| T2 | 세로 5목 | (7,4)~(7,8) 흑 | 승리 |
| T3 | 대각 ↘ 5목 | (3,3)~(7,7) 흑 | 승리 |
| T4 | 대각 ↗ 5목 | (7,3)~(3,7) 흑 | 승리 |
| T5 | 4목(미완성) | 연속 4 | 비승리 |
| T6 | 장목(6목) | 연속 6 | 승리(자유룰) |
| T7 | 좌상단 경계 | (0,0)~(0,4) | 정상 판정(배열 범위 안전) |
| T8 | 우하단 경계 | (14,10)~(14,14) | 정상 판정 |
| T9 | 끊긴 4+1 | 흑4 + 백 + 흑1 | 비승리 |

> 검증 방법: 콘솔에서 보드 배열을 세팅하고 `checkWin` 호출(임시 테스트 함수) 또는 실제 클릭으로 재현.

## 5. 우선순위 (MoSCoW)

| 구분 | 기능 |
|---|---|
| **Must** | FR-01 보드, FR-02 착수, FR-03 턴, FR-04 판정, FR-05 연출, FR-06 재시작 |
| **Should** | FR-07 무승부, FR-08 마지막 수, FR-09 무르기 |
| **Could** | FR-10 수순, FR-11 호버 |
| **Won't** | FR-12 사운드, AI, 금수룰, 온라인 |

## 6. 리스크 & 대응

| 리스크 | 영향 | 대응 |
|---|---|---|
| 5목 판정 경계 버그(배열 범위 초과) | 오작동/에러 | 카운트 루프에 0~14 범위 가드, §4 T7·T8 경계 케이스 검증 |
| 클릭 좌표 오차로 엉뚱한 칸 | 사용성 저하 | 픽셀→격자 변환 시 반올림 + 범위 검사 |
| 장목/승리 정책 모호 | 규칙 혼란 | 자유룰(5목 이상 승)로 명문화(SPEC §4) |
| 무르기 시 차례/상태 불일치 | 상태 깨짐 | `moveHistory` 단일 출처(SSOT)로 관리, undo는 pop 후 전체 redraw |
| 단일 파일 비대화 | 유지보수성 | 모듈 경계 유지, 필요 시 ARCHITECTURE §9 확장 구조로 분리 |
